# АКК Онбординг — демо-прототип

> **Что это.** Интерактивный прототип публичного онбординга АО «Аграрная кредитная корпорация»: подбор подходящей программы кредитования через 5–6 вопросов, калькулятор графика погашения и упрощённый стресс-тест экономики проекта.
>
> **Зачем.** Эталон для аналитиков и фронтенд-разработчиков, которые будут переносить эту логику на основной сайт [agrocredit.kz](https://agrocredit.kz). Здесь зафиксирована вся бизнес-логика подбора, расчётов и проверок — её достаточно перенести 1-в-1, без додумывания.
>
> **Прод-демо:** https://akk-production-4a6b.up.railway.app

## Источники правды

Все цифры, ставки, сроки и условия в `index.html` **должны соответствовать** двум документам:

1. **П АКК 002-207-22** — официальные нормативы и требования по продуктовым программам.

Когда регламент и сайт расходятся (а такое бывает) — **приоритет сайту**, потому что онбординг публичный, и пользователь будет сверять условия именно с сайтом.

## Тестирование (Playwright)

```bash
# Первый раз — поставить браузеры:
npx playwright install chromium

# Прогнать все тесты локально (поднимет сервер сам):
npm test

# Прогнать против прод-URL (без локального сервера):
npm run test:prod

# В headed-режиме (увидеть браузер):
npm run test:headed

# Открыть HTML-репорт после прогона:
npm run test:report
```

Что покрыто:

- **Лендинг:** 9 программ в превью-сетке (Кең дала временно скрыта — доступна только через посредников)
- **Квиз:** сквозные сценарии (ВПРиУР, племенной скот КРС 100–499 и 500+, микрокредит, откорм, инвестиции), кнопка «Назад». Аквакультура, Жайлау и тепличные хозяйства/сады показываются только в обзорной сетке (в авто-подбор квиза не входят — нет различающего вопроса)
- **Калькулятор:** двухплатёжный график Кең дала 2 (даты 05 декабря / 05 марта), ставка Игілік 6% / срок 108 мес, для Агробизнеса с целью ПОС срок ограничен 48 мес
- **Стресс-тест:** показ формы со селектором животного, вердикт после ввода данных
- **Объяснимость:** раскрывающийся разбор процента на карточке, в т.ч. недобор «до +N» при неполном совпадении

Тесты сделаны устойчивыми к мелким стилистическим правкам — селекторы по тексту кнопок/вариантов, а не по классам.

## Бизнес-логика

### Программы и параметры

Массив `PROGRAMS` в [index.html:1556](index.html#L1556) — **источник правды** для онбординга. Каждая программа описана единообразной структурой:

```js
{
  id: 'ken_dala_2',            // уникальный ключ, для аналитики и URL
  title: 'Кең дала 2',         // отображаемое название
  category: 'ВПРиУР',          // короткий тег для бейджа
  org: '...',                  // подзаголовок (откуда программа)
  description: '...',          // 1–2 предложения о программе
  rate: 5,                     // числовая ставка для калькулятора
  rateRange: '5–18%',          // (опц.) текстовый диапазон для отображения
  rateNote: '...',             // (опц.) пояснение про источники/каналы
  maxAmount: 1500000000,       // максимальная сумма, ₸
  maxTerm: 18,                 // максимальный срок, мес.
  termByPurpose: {             // (опц.) разные потолки срока по цели
    investments: 144,
    working: 48
  },
  scheduleType: 'biannual_winter' | 'annual',
  scheduleNote: '...',         // (опц.) пояснение про график
  hard: [{ q: 'purpose', v: ['vprir'] }, ...],   // hard-фильтры (AND)
  soft: [{ q: 'experience', v: ['3plus'], w: 40 }, ...],  // мягкие веса (OR внутри одного q)
  hasStressTest: true,         // (опц.) включает Fajr-lite шаг перед коллбэком
  indirectOnly: true           // (опц.) программа недоступна напрямую — только через КТ/МФО/БВУ
}
```

7 базовых программ соответствуют публичной странице agrocredit.kz дословно по ставке, сумме и сроку. Ещё 3 (Аквакультура, Жайлау, тепличные хозяйства/сады) заведены из регламента П АКК 002-130-21 и показываются только в обзорной сетке.

### Алгоритм подбора (scoring)

```
scoreProgram(program, answers):
  1. Для каждого hard-правила: если ответ пользователя НЕ в списке допустимых v —
     программа отсекается (return null).
  2. Для каждого soft-правила: если ответ пользователя в v — добавить вес w к score.
  3. score = min(100, sum_of_weights).
```

Программы с `null` (не прошли hard) не показываются. Остальные сортируются по score убыванию.

Порог для попадания в результаты — `score >= 20` (см. `scoredPrograms()`). Это отсекает совсем неподходящие варианты.

**При переносе:** этот алгоритм безбагово переносится в любую SPA — это чистая функция от двух объектов.

### Объяснимость процента совпадения

Чтобы пользователь понимал, **из чего сложился `score%` и почему он не 100%**, на карточке есть раскрывающийся разбор:

- **`explainProgram(program, answers)`** — разбор скоринга. Группирует `soft`-правила по вопросу (`softFactors`): для каждого вопроса `w` — сколько баллов реально дал ответ пользователя, `maxW` — максимум по этому вопросу. Если `w < maxW` — это и есть «недобор» до 100%. Также возвращает `passedHard`, `rawScore`, `score`. `scoreProgram()` — тонкая обёртка над ним. Подписи берутся из `label` вопросов квиза (`optionLabel`, `questionShort`) — единый источник, не разъезжается при правках.
- **Разбор на карточке** (`renderWhyMatch`) — раскрывашка «Почему N% совпадения»: каждый фактор профиля с его вкладом. Строки с пометкой «до +N» показывают, где недобор. Если все факторы максимальны, но `score < 100` — выводится пояснение, что это потолок программы для данного набора ответов.

**Важно для переноса:** `soft`/`score` — это индикатор «насколько программа близка профилю», а **не чек-лист**. Низкий процент не значит «сделай X» — частые причины недобора (опыт, отрасль) пользователь не может изменить по своей воле. Поэтому разбор показан как прозрачность цифры, а не как руководство к действию. Ворота допуска — это `hard`-условия (см. описание `scoreProgram` выше).

### Калькулятор графика

Две схемы:

1. **`biannual_winter`** (только Кең дала / Кең дала 2): два платежа — 50% после уборки урожая в декабре, 50% к марту. Проценты считаются на 8 месяцев на полную сумму + 3 месяца на остаток.
2. **`annual`** (все остальные): аннуитет по годам. Ежегодно платится фиксированная часть тела (`amount / years`) + проценты на остаток (`remaining * rate / 100`).

Формулы упрощены — это публичный pre-screen. Финальный график строит менеджер по бизнес-плану.

**Срок** для калькулятора берётся через `effectiveMaxTerm(p)`:
- Если у программы есть `termByPurpose` и пользователь выбрал цель — используется лимит по этой цели
- Иначе — общий `p.maxTerm`

Пример: Агробизнес имеет `maxTerm: 120` для ОС/СМР, но `termByPurpose.working: 48` для ПОС. Если пользователь выбрал «Пополнение оборотных средств», калькулятор покажет максимум 48 мес.

### Стресс-тест Fajr-lite

Только для программ с `hasStressTest: true` (сейчас — Игілік и Береке).

**Идея.** В полноценной финмодели Fajr (внутренний документ АКК, в публичный репозиторий не входит) есть три уровня:
1. Платёжный калькулятор — «сколько я отдам банку»
2. **Fajr-lite — «хватит ли ресурсов проекта»** ← это и есть наш стресс-тест
3. Полный Fajr — закрытый АРМ менеджера

Что считает Fajr-lite:

```
Ввод от пользователя:
  - вид животных (КРС / МРС / лошади / верблюды)
  - текущее поголовье
  - планируемое к покупке
  - пастбища, га
  - помещения, м²
  - текущие платежи по кредитам, тыс ₸/мес (упрощённая замена ПКБ-отчёта)
  - доп. доход от других направлений, млн ₸/год

Расчёт по нормативам FAJR_NORMS (заимствованы из FAJROptions полной Fajr):
  Доход = молоко (для КРС) + мясо + доп.доход
  Расход = (корма + ветеринария + прочие) × поголовье
  Чистый доход = Доход - Расход

  Совокупный годовой платёж = новый кредит + текущие × 12
  Ratio = платёж / чистый доход × 100%

Вердикт:
  - проверка пастбищ (га × норматив pastureHaPerHead)
  - проверка помещений (м² × минимум minBarnSqmPerHead)
  - проверка нагрузки на доход (ratio < 30% — комфортно, 30–50% — напряжённо, >50% — тяжело)
```

Нормативы (`FAJR_NORMS`) различаются по виду животных — это критично, потому что 1,5 га/голову подходит для КРС, но для лошадей это 3 га, для верблюдов 5 га.

## Что нужно учесть при переносе на основной сайт

### Для аналитиков

1. **Любое изменение в регламенте или на сайте → отрази в `PROGRAMS`** и обнови тест в `onboarding.spec.js`. Тесты должны падать, когда параметры изменились — это feature, не bug.
2. **Не упрощай rate до одного числа**, если на сайте указан диапазон или формула. Используй `rateRange` для отображения, `rate` — для калькулятора (минимум или типовое значение).
3. **Программы с `indirectOnly: true`** не должны попадать в результаты квиза для конечного пользователя — они только в обзорной сетке.

### Для фронтенд-разработчиков

При переносе на основной сайт (предположительно Angular/React/Vue) рекомендую разнести по модулям:

```
src/
├── data/
│   ├── programs.ts         ← массив PROGRAMS (1-в-1 из index.html)
│   ├── questions.ts        ← QUESTIONS + HEADS_Q
│   └── fajr-norms.ts       ← FAJR_NORMS
├── lib/
│   ├── scoring.ts          ← scoreProgram, scoredPrograms (чистые функции)
│   ├── schedule.ts         ← calculateSchedule, effectiveMaxTerm
│   ├── stress.ts           ← calculateStress
│   └── format.ts           ← fmtAmount, fmtRate, declension
├── components/
│   ├── Hero
│   ├── ProgramGrid
│   ├── Quiz (QuestionStep + Progress)
│   ├── Results (ResultCard + Calculator)
│   ├── StressTest
│   └── CallbackForm
└── analytics.ts            ← track() (заменить console.log на ваш трекер)
```

**Что важно сохранить:**

- **Единый стейт онбординга** — все ответы, выбранная программа, состояние стресс-теста, форма обратной связи. Не разносите по разным сторам.
- **Условные вопросы про скот** активируются по цепочке: вид животных и кол-во голов — когда `purpose === 'livestock'`. Не показывать всем подряд. (Вопрос про импортное/отечественное поголовье убран: по регламенту 002-207-22 Игілік/Береке финансируют племенной КРС и импортный, и отечественный.)
- **Прогресс-бар** учитывает реальное число шагов: 5 для большинства, 6 для покупки скота, 7 для покупки КРС. См. `getQuestions()`.
- **Аналитика на каждый шаг** — без неё непонятно, где пользователи отваливаются. Список events ниже.

**Чего НЕ делать:**

- ❌ Не превращайте калькулятор в форму с типом ОС/ПОС/Облигационный заём — это публичный pre-screen, лишний выбор отпугнёт. Этот вопрос задаёт менеджер на этапе экспертизы.
- ❌ Не показывайте полную форму Fajr (с поголовьем по видам, видами кормов, площадями по типам). Уровень 1–2 — это просто индикатор «хватит — не хватит». Полный Fajr живёт в АРМ менеджера.
- ❌ Не валидируйте телефон жёстко. Достаточно базовой маски `+7 (___) ___-__-__`. Любые символы кроме цифр менеджер всё равно отбросит.

### Для бэкенда (если будет API)

Onboarding в текущей форме — фронт-only. Когда появится бэкенд:

- `POST /api/leads` — отправить заполненный коллбэк. Поля: `program`, `name`, `phone`, `channel`, `time`, `answers` (все ответы квиза), `calc` (сумма+срок), `stress` (если был).
- Все события из `track()` — должны лететь в аналитический эндпойнт.
- Список программ можно отдавать с бэка (CMS), но **тогда нужен миграционный план** — какие поля обязательны, что nullable. Структура из `PROGRAMS` — рабочий контракт.

## Аналитика и events

Каждый шаг трекается через `track(name, payload)`. Список событий:

| Event | Когда | Payload |
|---|---|---|
| `quiz_started` | пользователь нажал «Начать подбор» | — |
| `quiz_answer` | выбран ответ на вопрос | `{ question, answer, step }` |
| `quiz_completed` | пройдены все вопросы | `{ answers }` |
| `results_viewed` | показан экран результатов | `{ count, top, top_score }` |
| `calculator_amount` | изменена сумма в калькуляторе | `{ program, amount }` |
| `calculator_term` | изменён срок | `{ program, term }` |
| `apply_clicked` | нажата «Подать заявку» | `{ program, score, amount }` |
| `stress_test_opened` | открыт стресс-тест (только Игілік/Береке) | `{ program }` |
| `stress_test_completed` | стресс-тест посчитан | `{ program, animal_type, ratio, verdict, total_herd, has_existing_debts }` |
| `stress_test_skipped` | пользователь пропустил стресс | `{ program }` |
| `callback_form_opened` | открыта форма обратной связи | `{ program }` |
| `lead_submitted` | форма отправлена | `{ program, channel, time }` |

В демо `track()` пишет в `console.log` и в правую боковую панель (для презентации). В проде — заменить на отправку в GA4 / Яндекс.Метрику / собственный коллектор.
